## What Karpathy's nanoGPT Actually Is

nanoGPT trains a GPT-2 style transformer from scratch — ~124M parameters — on next-token prediction. The magic is it's ~300 lines of Python, no dependencies, fully understandable.

The key insight: it works because GPT-2 scale is enough for the task it's trained on (Shakespeare, code, etc).

## The Honest Problem for PSM

Training from scratch requires the model to learn:

```
Language understanding     ← needs massive pretraining data
        +
JSON structured output     ← needs the above first
        +
Memory operation reasoning ← what we actually want
```

Karpathy trained on billions of tokens of text to get language understanding. We have 750 training examples. A model trained from scratch on 750 examples learns nothing — it just memorizes them and fails on anything new.

Fine-tuning Qwen works because Qwen already learned language from trillions of tokens. We only teach it the narrow memory operations layer on top.

## What a Nano PSM From Scratch Would Look Like

```python
# Not GPT-2 style autoregressive
# But a small encoder-only transformer
# Like BERT, not GPT

class NanoPSM(nn.Module):
    def __init__(self):
        # Tiny encoder — just understand the input
        self.encoder = TransformerEncoder(
            d_model = 256,      # tiny
            nhead = 4,
            num_layers = 4,     # 4 layers is enough
            dim_feedforward = 512
        )
        
        # Classification head — what operation?
        self.operation_head = nn.Linear(256, 10)  # 10 operation types
        
        # Extraction heads — what are the memory fields?
        self.strength_head = nn.Linear(256, 1)
        self.decay_head = nn.Linear(256, 1)
        self.emotional_head = nn.Linear(256, 1)
        self.type_head = nn.Linear(256, 3)  # episodic/semantic/archival
        self.tier_head = nn.Linear(256, 3)  # which tier to search
```

Total parameters: **~2-5M**. Runs in **milliseconds on CPU**. No GGUF, no quantization issues, no JSON parsing problems.

## The Architecture

```
Input tokenization (simple BPE or even character-level)
        ↓
Conversation turns + memory store → fixed-length encoding
        ↓
4-layer transformer encoder (256 dim)
        ↓
┌─────────────────────────────────┐
│  Operation classifier           │ → store_episodic / ignore / promote...
│  Content extractor (span-based) │ → which text to store
│  Strength regressor             │ → 0.0-1.0
│  Decay regressor                │ → 0.0-1.0  
│  Emotional weight regressor     │ → 0.0-1.0
│  Type classifier                │ → episodic/semantic/archival
│  Tier selector (for retrieval)  │ → which tables to search
└─────────────────────────────────┘
```

Each head is trained independently with its own loss. Operation classification uses cross-entropy. Regressors use MSE. Content extraction uses span prediction like SQuAD.

```json
  }
}
```

Our 750 examples already have all this information — just need to reformat them into classification labels instead of JSON generation targets.

**Training:** Single GPU, 30 minutes. Or even CPU training since the model is tiny.

**The nanoGPT spirit** — the whole thing fits in one file, ~400 lines of PyTorch.

## The One Real Challenge

**Content extraction is harder than classification.**

Deciding "what text to store as the memory content" requires either:
- Span extraction (like SQuAD) — extract the relevant span from the input
- Abstractive summarization — generate a short summary

Span extraction works without any generation but limits you to storing exact phrases from the input. For most PSM use cases that's fine — "Melanie ran a charity race on May 21 2023" is right there in the input.

For the cases where you need abstraction — "User is a Python developer" synthesized from many mentions — you keep a tiny generative head just for content, maybe 50M params. Everything else stays classification.
