## Problem diagnosis (why gates pass but LoCoMo fails)

### 1. Training distribution ≠ deployment task

Gate eval probes look like:

```json
{"conversation": "User: For future coding tasks, always gate datasets before training.", "operation": "remember"}
```

LoCoMo / prod plain text (after `to_model_input`):

```text
User: Caroline said "I went to a LGBTQ support group yesterday and it was so powerful."
```

058000 scores **99%+** on the first family and **fails** on the second — same checkpoint, same code path.

### 2. Curriculum dominance → OOD regression

Much of Gate 4/5 training mass is **PSM-internal**:

- gate datasets, checkpoints, direct probes, PowerShell, RunPod, tagged DSL drills  
- Generated by `generate_direct_behavior_curriculum.py`, expanded probes, fact-format drills

When the model sees OOD chat input, a 50M LM often **completes with memorized curriculum tokens** (garbage like lanes, validation, checkpoint counts). That is regression to the highest-density training patterns, not random noise.

### 3. LoCoMo dialogue is not the same as “train on LoCoMo”

Some nano-blend rows contain Caroline/Melanie chunks, but often as:

- multi-turn `Caroline:\nMelanie:` blocks (not `User: Speaker said …`)
- labels like `ignore` for whole segments
- rows in **test** split of expanded probe, not the main training anchor

We need **more store/ignore/conflict examples** for **single durable facts from natural dialogue** in **prod inference shape**. We must **not** add LoCoMo QA labels or eval turns to training.

### 4. What we verified experimentally

| Test | Result |
|------|--------|
| In-distribution gate probe on CPU/CUDA | **Correct** `C:` line (e.g. “gate datasets before training”) |
| D1:3 plain product text | **Garbage** `C:` line (CPU and CUDA) |
| Plain vs narrative LoCoMo ingest | Plain removes curriculum-bleed flags; gold facts still missing |
| Product prompt envelope | Already normalized in `to_model_input` — not the blocker |

## Training row schema (required shape)

Every new row must be a valid training JSONL object:

```json
{
  "id": "conv-mem-unique-id",
  "input": {
    "conversation": "User: Caroline said she went to an LGBTQ support group yesterday and it was powerful.",
    "operation": "remember",
    "source_kind": "synthetic_dialogue",
    "source_id": "session-1:turn-3",
    "source_timestamp": "1:56 pm on 8 May, 2023"
  },
  "expected": {
    "action": "store_episodic",
    "memory": {
      "content": "Caroline went to an LGBTQ support group yesterday and found it powerful.",
      "type": "episodic",
      "strength": 0.86,
      "decay_rate": 0.02,
      "emotional_weight": 0.35,
      "confidence": 0.92,
      "tags": ["event", "lgbtq"],
      "temporal_expression": "yesterday",
      "resolved_time": "7 May 2023"
    },
    "facts": [],
    "reasoning": "The speaker stated a durable personal event with relative time."
  },
  "source": "gate6:synthetic-dialogue",
  "split": "train"
}
```

### Input rules (prod alignment)

| Rule | Detail |
|------|--------|
| **Prefix** | `conversation` must be a single string starting with `User: ` for storage rows |
| **Plain text** | Natural language; no benchmark metadata (`Current utterance:`, `Source id:`, JSON blobs) |
| **Speaker names** | Use real names from dialogue in `memory.content` — not generic “User” when a name is known |
| **Relative time** | If utterance has yesterday/last week, set `temporal_expression` + `resolved_time` when `source_timestamp` anchors it |
| **operation** | `"remember"` on all storage rows |
| **Tagged output** | Training target uses tagged DSL (`output_format: tagged`) — same as 058000 |

### Output / label rules

| Action | When |
|--------|------|
| `store_episodic` | Durable event, statement, image caption fact |
| `promote_semantic` | Stable preference/rule stated explicitly |
| `ignore` | Greetings, backchannel, questions without new facts, pure logistics |
| `flag_conflict` | Contradicts `context.memory_store` in input |
| `update_existing` | Refinement of same entity/fact |

**memory.content** must be a **concise extraction**, not a copy of the wrapper or full transcript.

Validate every row: `validate_training_row()` + `validate_storage_decision(expected)`.

## What training data we need (new row families)

Goal: cover the **deployment task** without touching LoCoMo labels.

### Family A — Single-turn dialogue store (highest priority)

**What:** One memorable utterance per row.

```text
User: Melanie said she painted a sunrise over a lake last weekend.
User: Caroline said she is interested in counseling and mental health work.
```

**Labels:** `store_episodic` or `promote_semantic` with grounded `memory.content` and optional facts.

**Volume target:** 3,000–8,000 unique rows (before copy multiplier).

### Family B — Single-turn ignore / noise

**What:** Greetings, “sounds good”, questions, filler.

```text
User: Caroline said "Hey Mel! Good to see you! How have you been?"
User: Melanie said "What's up with you? Anything new?"
```

**Labels:** `ignore`, `memory: null`.

**Volume target:** 2,000–4,000 rows (model over-stores without these).

### Family C — Multi-turn context, single fact target

**What:** 2–4 lines of `Speaker: text` joined with `\n` in the User string (nano-style), but normalized to prod evaluation path.

```text
User: Caroline: I went to a support group yesterday.
Melanie: Wow, tell me more!
Caroline: The transgender stories were inspiring.
```

**Labels:** Often `ignore` for low-signal windows; `store_episodic` only when the marked turn has a durable fact. Prefer **mining** from nano with human/teacher labels already present.

**Volume target:** 2,000–5,000 rows.

### Family D — Conflict / update with memory_store context

**What:** `input.context` includes existing memory JSON; new turn contradicts or refines.

**Source:** Extend `generate_direct_behavior_curriculum.py` patterns but with **chat content** (preferences, plans), not only checkpoint rules.

**Volume target:** 500–1,500 rows.

### Family E — Image / caption turns

**What:** Utterance + image query/caption in plain text (matches `buildLocomoProductText`).

```text
User: Caroline said "The transgender stories were inspiring!". Image query: transgender pride flag mural. Image caption: a photo of a dog walking past a wall with a painting of a woman.
```

**Volume target:** 500–1,500 rows.

### Family F — Real chat exports (diversity)

**What:** Convert external chat logs with weak or LLM-assisted labels.

**Sources (existing tooling):**

| Tool | Path | Notes |
|------|------|--------|
| Nano converter | `psm-model/src/psm_model/convert_nano_dataset.py` | Has `Caroline: text` — **must normalize to `User: …` prod shape** |
| ChatGPT exports | `psm-model/src/psm_model/convert_chatgpt_exports.py` | User-turn heuristics for preferences/rules |
| Stratified sampling | `build_gate4_train_v3.py` `_sample_stratified_rows` | Real chat diversity pattern |

**Volume target:** 2,000–10,000 rows after filtering (drop invalid schema).

### Explicitly exclude from train

- `benchmark/locomo/data/locomo10.json` turns used in current eval (`--limit 25` or full)
- Any row whose `id` appears in LoCoMo ingest debug `source` fields for eval runs
- Codex sessions explaining how LoCoMo benchmark works (meta bleed — already in expanded probe test split)

## How to generate the data (concrete pipelines)

### Pipeline 1 — Synthetic dialogue generator (new script recommended)

**Create:** `psm-model/src/psm_model/generate_conversation_memory_curriculum.py`

**Approach:**

1. Template pools: speakers, life events, preferences, careers, family, medical, hobbies, travel.
2. For each sample:
   - Pick action (weighted: 40% store, 35% ignore, 15% promote, 10% conflict).
   - Emit `input.conversation` as `User: {speaker} said {quoted utterance}.`
   - Emit `expected` with concise `memory.content` (not verbatim transcript).
   - Add `source_timestamp` when using yesterday/last week.
3. Deterministic seeds for reproducibility.
4. Write JSONL + summary stats (action counts, avg content length).
5. CLI: `python -m psm_model.generate_conversation_memory_curriculum --rows 5000 --out psm-model/data/curriculum/conversation-memory-synthetic-v1.jsonl`

**Quality filters:**

- Reject if `memory.content` contains: `checkpoint`, `PowerShell`, `gate datasets`, `nvidia-smi`, `StorageDecision`
- Reject if content length > 200 chars for episodic (force concision)
- Reject if `User:` missing from conversation

### Pipeline 2 — Nano blend normalization

**Existing:** `convert_nano_dataset.py` → `canonical_input()` produces `Caroline: text\nMelanie: text`.

**Add normalization step** (function or post-processor):

```python
def to_prod_conversation(conversation: str) -> str:
    if conversation.startswith("User:"):
        return conversation
    # Single-speaker single line → User: Speaker said "..."
    # Multi-line Speaker: → User: + joined block OR split into single-turn rows
```

**Split strategy:** Prefer **one training row per durable turn** rather than long multi-turn blocks labeled `ignore` — aligns with prod `remember()` per turn.

**Command pattern:**

```bash
python -m psm_model.convert_nano_dataset \
  --input-dir /path/to/nano-jsonl \
  --out psm-model/data/curriculum/conversation-memory-nano-v1.jsonl \
  --normalize-prod-conversation
```

(Flag to be implemented.)

### Pipeline 3 — ChatGPT export mining

```bash
python -m psm_model.convert_chatgpt_exports \
  --input-dir psm-model/data/raw/chatgpt-exports \
  --out psm-model/data/curriculum/conversation-memory-chatgpt-v1.jsonl
```

Review action distribution; oversample `store_episodic` if too preference-heavy.

### Pipeline 4 — Holdout hygiene


## Evaluation strategy (training gates + LoCoMo)

### Must pass (block ship)

| Eval | Command / path | Threshold |
|------|----------------|-----------|
| Dual gate | `python -m psm_model.eval_dual_gate <ckpt> --device cuda` | `passed: true` |
| Expanded storage | included in dual | action ≥0.85, content exact ≥0.50, parse ≥0.95 |
| Recall plan | included in dual | parse ≥0.95, top_k exact ≥0.90 |

### Should improve (track, do not train on)

| Eval | Command | Target |
|------|---------|--------|
| LoCoMo quality | `run-locomo-psm-model.ps1 -Limit 25 -Device cpu -Step <id> -SkipAnswer` | `ingest-quality-check` **passed: true** |
| LoCoMo gold probes | D1:3, D1:5, D1:12 in quality JSON | facts present in stored content |
| LoCoMo retrieval | `evaluate.js` | Hit@3 trending up (secondary) |
| LoCoMo answers | `answer-evaluate.js` | Only after quality passes; needs `OPENROUTER_API_KEY` |

### CPU vs CUDA note

- Gate eval on RunPod: **CUDA**
- Local LoCoMo smoke: **CPU** with `PSM_FORCE_CPU=1` (acceptable for trend, slower)
- In-distribution probes work on CPU; OOD LoCoMo fails on both — training fix is not CPU-specific
